# 结构化教练式训练流程设计规格

**日期**：2026-06-24  
**状态**：待用户复核，未进入实施计划  
**适用范围**：`studyGo/` 单人本地训练闭环  
**风险级别**：L3，仅规格确认；实施前需再次人工确认

## 1. 目标

将当前“初稿 → AI 诊断 → 二次改写 → 结果复盘”的批改型闭环，升级为“初稿 → AI 初诊断 → 结构化追问训练 → 最终复述 → 前后对比”的教练式闭环。

核心目标：

- 让训练价值发生在用户被追问、补充、重组表达的过程中，而不是只看一次反馈。
- 保持 AI 的角色为诊断、追问、反馈和对比，不替用户代写完整答案。
- 固定轮次与输出契约，避免产品退化为无限自由聊天。

## 2. 非目标

- 不做自由聊天入口。
- 不做完整范文、整段润色稿或替用户完成最终表达。
- 不引入登录、云同步、多用户、语音或付费能力。
- 不改变 Provider 安全边界、API Key 存储规则或本地优先原则。
- 不在本规格阶段修改代码、数据结构或 Prompt。

## 4. 推荐流程

目标状态机：

```text
setup → topic → draft → diagnosis → coaching → finalRewrite → result
```

阶段说明：

1. `setup`：用户选择场景、难度和训练目标，沿用现有能力。
2. `topic`：AI 生成结构化命题，沿用现有能力。
3. `draft`：用户写 200-400 字初稿，AI 暂不介入。
4. `diagnosis`：AI 初诊断，只识别最大短板和本次追问路径，不给完整改写。
5. `coaching`：AI 进行固定 3 轮结构化追问，每轮只训练一个目标。
6. `finalRewrite`：用户基于三轮追问自己完成最终复述，仍为 200-400 字。
7. `result`：系统展示初稿、最终复述、追问过程、评分变化、剩余短板和下一练。

## 5. 三轮教练结构

MVP 固定 3 轮，避免无限聊天：

| 轮次 | 训练目标 | AI 追问方向 | 达标信号 |
| --- | --- | --- | --- |
| 第 1 轮 | 立场与边界 | 观点在什么条件下成立，是否有例外 | 用户说出至少一个判断条件或边界 |
| 第 2 轮 | 理由与证据 | 有什么事实、经验、例子或机制支撑 | 用户给出至少一个具体支撑材料 |
| 第 3 轮 | 结构与表达 | 如何组织顺序让别人更容易听懂 | 用户能说明表达顺序或取舍 |

每轮交互：

```text
AI 追问
→ 用户回答
→ AI 判断是否推进本轮目标
→ AI 简短反馈
→ 达标则进入下一轮，不达标则降阶引导
```

## 6. 回答不如预期时的处理

追问阶段不能假设用户一定答好。用户回答不充分时，系统进入“降阶引导”，而不是失败或代写。

规则：

```text
主追问 1 次
→ 用户回答
→ 若不达标，补救追问 1 次
→ 若仍不达标，补救追问 2 次
→ 仍不达标，记录本轮短板并继续推进
```

补救追问允许：

- 缩小问题范围。
- 给用户可选方向，例如“储蓄、机会、家庭压力、心理状态里选一个”。
- 要求用户把模糊词说具体，例如“有钱”改成“几个月生活费”。
- 总结用户已说出的局部材料，并提示缺口。

补救追问禁止：

- 给出完整标准答案。
- 替用户写出最终复述段落。
- 把用户观点改成 AI 自己的立场。
- 因立场不同判定不达标。

不达标后的推进原则：

- 最多补救 2 次，避免卡关。
- 仍不达标时，进入下一轮，但在结果页标记该轮仍是短板。
- 最终复述前生成“待补清单”，只引用用户已有回答和本轮缺口，不生成成稿。

## 7. AI 输出边界

AI 在新流程中只允许输出：

- 初诊断：一句话总评、最大短板、追问路径。
- 追问：一个聚焦问题。
- 轮次反馈：本轮是否推进、已补出的材料、下一步缺口。
- 补救引导：更小的问题或可选方向。
- 最终复述前任务清单：用户需要补清的点。
- 结果复盘：前后对比、分数变化、剩余问题和下一练。

AI 不允许输出：

- 完整范文。
- 完整最终复述。
- 整段替用户润色后的答案。
- 对用户立场本身的价值评判。

## 8. 结果复盘

结果页以“用户自己的变化”为中心：

- 初稿原文。
- 三轮追问记录。
- 每轮是否达标与补救次数。
- 最终复述。
- 八个维度分数变化。
- 聚合逻辑分、表达分、提升值、覆盖数和最低维度。
- 本次仍未补清的短板。
- 下一练推荐。

聚合分、提升值、覆盖数和最低维度继续由服务端或确定性函数重算，不能信任模型直接给出的聚合结果。

### 9.8 关键交互约束

- 所有 AI 和用户内容按普通文本渲染，禁止 `dangerouslySetInnerHTML`。
- 追问阶段主任务区使用聊天气泡上下堆叠，保留上下文因果链，不使用左右分栏表单承载当前问答。
- 追问阶段底部固定一个主要输入框，避免用户同时处理多轮问题。
- “记录为短板并继续”只在补救 2 次后出现，避免用户过早跳过训练。
- 最终复述提交前必须满足 200-400 grapheme cluster 计数。
- 结果未保存时仍沿用现有离开保护，不允许重复调用 AI 规避保存失败。

## 10. 数据保存

训练全文仍只进入 IndexedDB，不进入 localStorage。

建议训练记录新增：

- `coachingRounds`：每轮目标、追问、用户回答、反馈、达标状态、补救追问。
- `finalRewriteText`：用户最终复述文本。
- `coachingSummary`：本次教练过程摘要。
- `unresolvedWeaknesses`：仍未补清的短板。

API Key 规则不变：只允许存在于版本化 Provider 设置和发往本地 Route Handler 的 JSON 请求体，不写入训练记录、日志、URL、截图或测试快照。

## 11. 错误与恢复

- AI 请求前必须保存当前会话，包括追问轮次和用户回答。
- 请求失败时停留当前轮次，保留用户文本，允许重试。
- 重复请求必须取消旧请求；旧响应不得覆盖新轮次或新文本。
- 刷新后恢复到最近保存的轮次，而不是重新生成追问。
- 进入 `result` 后保存失败只能重试保存，不能重新调用 AI 生成结果。

## 12. 验证要求

实施时需要先补失败测试，再改代码。

局部验证至少覆盖：

- Schema 拒绝缺轮次、重复维度、完整代写字段和非法分数。
- Store 能保存每轮回答、补救次数、最终复述，并拒绝旧响应写回。
- Mock Provider 产出稳定的三轮教练结果。
- 真实 Provider 输出经过 Zod 校验和一次 JSON 修复重试。
- Repository 能保存并恢复教练轮次。
- UI 能在桌面和移动视口完成完整闭环。

完整验证按影响范围运行：

```text
pnpm lint
pnpm typecheck
pnpm test
pnpm build
pnpm test:e2e
```

## 13. 回滚点

推荐把实现拆成可回滚阶段：

1. 只新增 Schema、类型和 Mock Provider 能力，不接入 UI。
2. 接入 Store 和 Repository，但保留旧流程可运行。
3. 接入新 UI 流程和结果页。
4. 移除或迁移旧“二次改写”主流程。

若教练流程体验不达预期，可回滚到旧状态机：

```text
setup → topic → draft → diagnosis → result
```

并保留本规格作为后续迭代依据。
